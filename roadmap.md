# Roadmap Android-порта Digger

## Цель проекта

Сделать Android-порт Java-версии Digger с сохранением оригинального ощущения игры: пиксельная CGA-графика 320x200, фиксированный игровой темп, классические уровни, мешки, монстры, бонусы, очки и звук.

Первый релиз должен быть небольшим, но играбельным. Главная цель MVP - пройти путь от запуска приложения до полноценной игровой сессии на одном уровне с управлением, столкновениями, смертью, завершением уровня и паузой.

## Базовые решения

- Язык приложения: Kotlin для Android-слоя.
- Ядро игры: Java или Kotlin без зависимостей от Android API.
- Рендеринг: `SurfaceView` или `TextureView` с внутренним буфером 320x200.
- Масштабирование: nearest-neighbor с сохранением пропорций и letterbox-полями.
- Игровой цикл: фиксированный timestep, совместимый с текущим `frametime`.
- Управление: виртуальный D-pad, кнопка огня, кнопка паузы.
- Звук: сначала событийные эффекты, затем приближение к PC Speaker-музыке.
- Минимальная версия Android: выбрать после создания проекта, ориентир - современный LTS-диапазон Android Gradle Plugin.

## Архитектура

Проект стоит разделить минимум на два модуля:

- `:core` - платформенно-независимая логика игры.
- `:app` - Android UI, рендеринг, ввод, звук, lifecycle и настройки.

Основные интерфейсы между слоями:

- `GameInput` - состояние команд игрока: направление, огонь, старт, пауза.
- `GameRenderer` - операции рисования в виртуальный экран 320x200.
- `AudioDriver` - звуковые события и управление музыкой.
- `Clock` или `GameLoop` - контроль времени кадра и паузы.
- `PersistentStorage` - рекорды, настройки, сохранение состояния.

## Этап 0. Инициализация Android-проекта

Задачи:

- Создать Gradle-проект Android.
- Добавить модули `:app` и `:core`.
- Настроить Kotlin, Android Gradle Plugin, namespace и базовый `minSdk`.
- Добавить пустой `MainActivity`.
- Подключить простую проверку сборки в CI или локальную команду.

Критерии готовности:

- `./gradlew assembleDebug` успешно собирает приложение.
- Приложение запускается на эмуляторе или устройстве.
- В репозитории есть понятная структура модулей.

## Этап 1. Каркас игрового экрана

Задачи:

- Создать `GameActivity` или один `MainActivity` с игровым экраном.
- Добавить `SurfaceView`/`TextureView`.
- Реализовать игровой поток с фиксированным timestep.
- Нарисовать тестовый кадр 320x200 и масштабировать его на экран.
- Добавить letterbox-поля и nearest-neighbor масштабирование.
- Обработать `onPause`, `onResume`, остановку и перезапуск игрового потока.

Критерии готовности:

- На экране стабильно отображается тестовый пиксельный кадр.
- При повороте экрана и смене размера изображение сохраняет пропорции.
- При сворачивании приложения игровой поток корректно останавливается.

## Этап 2. Выделение платформенно-независимого ядра

Задачи:

- Проанализировать текущие классы Java-версии:
  - `Digger`
  - `Main`
  - `Bags`
  - `Monster`
  - `Scores`
  - `Drawing`
  - `SpriteEngine`
  - `CgaDisplay`
  - `Input`
  - `Sound`
- Разделить код на игровую логику и платформенные зависимости.
- Убрать прямые зависимости ядра от AWT, `Frame`, `Canvas`, `KeyEvent`, Java Sound и `Preferences`.
- Описать минимальную модель состояния игры.
- Перенести данные уровней и правила перехода между уровнями.

Критерии готовности:

- Модуль `:core` компилируется без Android API и AWT.
- Игровой tick можно вызвать из теста или простой консольной проверки.
- Состояние игры обновляется без создания окна, canvas или audio output.

## Этап 3. Рендеринг оригинальной графики

Задачи:

- Перенести CGA-таблицы, спрайты, шрифт и title screen.
- Реализовать Android-версию виртуального экрана 320x200.
- Перенести операции:
  - очистка экрана;
  - рисование спрайта;
  - рисование спрайта с маской;
  - чтение пикселей для коллизий;
  - рисование текста;
  - переключение интенсивности палитры.
- Отрисовать title screen и статическое поле уровня.

Критерии готовности:

- Title screen визуально похож на Java-версию.
- Игровое поле первого уровня отображается без искажений.
- Пиксель-арт остается резким на разных размерах экрана.

## Этап 4. Управление

Задачи:

- Добавить экранные кнопки направления.
- Добавить кнопку огня.
- Добавить кнопку паузы.
- Сопоставить Android-ввод с командами ядра.
- Поддержать удержание направления, чтобы поворот срабатывал при выравнивании по сетке.
- Добавить базовую поддержку аппаратной клавиатуры или геймпада, если это не усложнит MVP.

Критерии готовности:

- Digger управляется с экрана без заметной задержки.
- Удержание направления работает так же удобно, как в Java-версии с клавиатурой.
- Огонь и пауза доступны во время игры.

## Этап 5. Играбельный вертикальный срез

Задачи:

- Запуск игры с title screen.
- Инициализация первого уровня.
- Движение Digger.
- Копание тоннелей.
- Сбор изумрудов и золота.
- Движение мешков.
- Движение монстров.
- Коллизии, смерть, потеря жизни.
- Завершение уровня.
- Возврат к title screen после game over.

Критерии готовности:

- Можно начать новую игру и играть первый уровень.
- Основные правила совпадают с Java-версией.
- Нет зависаний при смерти, завершении уровня и паузе.

## Этап 6. Звук

- Ввести интерфейс `AudioDriver` в ядре.
- Подключить Android-реализацию через `AudioTrack`, `SoundPool` или подготовленные короткие samples.
- Реализовать основные эффекты:
  - выстрел;
  - взрыв;
  - сбор изумруда;
  - падение мешка;
  - смерть;
  - бонус;
  - завершение уровня.
- Добавить переключение звука в настройках.
- Отдельно оценить перенос фоновой PC Speaker-музыки.

Критерии готовности:

- События игры сопровождаются звуками.
- Звук корректно останавливается при паузе приложения.
- Выключение звука сохраняется между запусками.

Критерии готовности:

- Рекорды не теряются после перезапуска приложения.

Критерии готовности:

- Сборка и тесты запускаются одной командой.

## Рекомендуемый порядок первых задач

1. Создать Android Gradle-проект.
2. Добавить `:core` и `:app`.
3. Поднять пустой игровой экран с `SurfaceView`.
4. Реализовать виртуальный экран 320x200 и масштабирование.
5. Перенести CGA-палитру и title screen.
6. Начать отделять игровую логику от AWT в исходном Java-проекте.
7. Перенести минимальный игровой tick в `:core`.
8. Подключить управление и получить первый играбельный уровень.

## Риски и решения

- Текущий `Digger` совмещает окно, цикл игры, ввод, звук и логику.
  Решение: переносить не класс целиком, а постепенно вытаскивать зависимости через интерфейсы.

- Коллизии завязаны на чтение пикселей из CGA-буфера.
  Решение: сохранить виртуальный пиксельный буфер в ядре или в общем rendering-слое, чтобы правила не разошлись с оригиналом.

- PC Speaker-звук может занять много времени.
  Решение: для MVP сделать событийные звуки, а точную эмуляцию музыки вынести в отдельный этап.

- Мобильное управление может изменить ощущение игры.
  Решение: как можно раньше тестировать D-pad на устройстве и настраивать размеры, зоны касания и поведение удержания.

- Десктопный код содержит много mutable state.
  Решение: сначала переносить поведение малыми кусками и покрывать его тестами, не пытаться сразу переписать архитектуру целиком.

## Definition of Done для MVP

- Приложение запускается на Android-устройстве.
- Title screen отображается.
- Можно начать игру.
- Первый уровень играбелен.
- Работают движение, копание, огонь, мешки, монстры, очки, смерть и завершение уровня.
- Есть пауза.
- Есть базовый звук или возможность выключить звук, если звук еще экспериментальный.
- Приложение корректно обрабатывает `onPause` и `onResume`.
- README содержит команды сборки и краткое описание управления.
